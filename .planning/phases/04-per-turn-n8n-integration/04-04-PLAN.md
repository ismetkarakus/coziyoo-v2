# Plan 04-04: End-to-End Per-Turn Flow Verification

## Goal

A real voice session demonstrates that mobile speech flows through STT, then n8n, and the voice agent speaks the `replyText` back — for multiple turns — within the 60-second n8n timeout window.

## Context

Plans 04-01 through 04-03 make structural fixes (required secret, single URL path, better error logging). This plan verifies the full per-turn pipeline works end-to-end on a real device against a live n8n instance.

The per-turn flow (per research):
1. Mobile user speaks → Silero VAD detects end of utterance
2. Voice agent STT provider transcribes audio → `userText`
3. `N8nLLMStream._run()` POSTs `{ workflowId, roomId, jobId, deviceId, userText, messages, systemPrompt, locale }` to the resolved webhook URL
4. n8n `brain_6KFFgjd26nF0kNCA` workflow processes the turn → `respondToWebhook` node returns `{ "replyText": "..." }`
5. Voice agent parses `replyText`, synthesises TTS audio, publishes to LiveKit room
6. Mobile user hears the response

**Session identity:** The voice agent sends `roomId` (stable for session lifetime) on every turn. n8n can use `roomId` as a session key. No separate `sessionId` field exists; `roomId` is the correct identifier.

**This plan contains no code changes.** It is a structured test checklist that an executor runs after Plans 04-01–04-03 are complete.

## Requirements Addressed

| Req ID | Description |
|--------|-------------|
| TURN-01 | Voice agent sends each user turn to n8n with session ID; n8n maintains session state across turns |
| TURN-02 | N8N webhook URL resolves through a single consistent path |
| TURN-03 | N8N workflow uses "Respond to Webhook" node and returns a reply before the 60-second timeout |
| TURN-04 | Voice agent correctly parses `{ "replyText": "..." }` from n8n response and speaks it back |
| TURN-05 | AI_SERVER_SHARED_SECRET validated at startup |

## Tasks

### Wave 1 — Pre-flight checks before live test

> Prerequisite: Plans 04-01, 04-02, and 04-03 are fully applied and services are running.

#### Task 04-04-01: Verify startup behaviour and environment

**What:** Confirm both the API and voice agent start cleanly with `AI_SERVER_SHARED_SECRET` set, and that the API logs the resolved n8n config at startup.

**Checklist:**

1. Confirm `AI_SERVER_SHARED_SECRET` is set in root `.env` (min 16 chars):
   ```bash
   grep "AI_SERVER_SHARED_SECRET" .env
   ```

2. Start the API and check the startup log for the n8n config line (added in plan 04-02):
   ```bash
   npm run dev:api 2>&1 | grep "n8n config"
   ```
   Expected: `n8n config: host=<url> webhookUrl=<url or "(not set)"> webhookPath=... workflowId=6KFFgjd26nF0kNCA`

3. Start the voice agent worker and confirm it does NOT exit immediately:
   ```bash
   cd apps/voice-agent && python -m voice_agent.entrypoint --help 2>&1 | head -5
   ```
   (The `--help` flag should print usage without triggering the actual worker loop, confirming the module loads without crash.)

4. Start the voice agent join API and confirm it binds:
   ```bash
   cd apps/voice-agent && uvicorn voice_agent.join_api:app --port 9000 --timeout-graceful-shutdown 1 &
   sleep 2
   curl -s http://localhost:9000/health | grep -i "ok\|healthy\|200" || curl -s -o /dev/null -w "%{http_code}" http://localhost:9000/health
   kill %1 2>/dev/null
   ```

**Verify:** All four checks pass without error messages. The API logs a non-empty n8n config line.

#### Task 04-04-02: Confirm n8n workflow is active and reachable

**What:** Verify the `brain_6KFFgjd26nF0kNCA` workflow is activated in the n8n UI and manually curl the webhook to confirm it responds with `{ "replyText": "..." }`.

**Checklist:**

1. Log in to the n8n instance (URL from `N8N_HOST` in `.env`).

2. Navigate to the workflow named "Brain" or with ID `6KFFgjd26nF0kNCA`. Confirm its status is **Active** (toggle is green). If it is inactive, activate it.

3. Identify the webhook URL. It should be:
   - If `N8N_LLM_WEBHOOK_URL` is set in `.env`: use that value
   - Otherwise: `{N8N_HOST}/webhook/6KFFgjd26nF0kNCA`

4. Send a test turn via curl:
   ```bash
   N8N_WEBHOOK_URL=$(grep "^N8N_LLM_WEBHOOK_URL" .env | cut -d= -f2-)
   if [ -z "$N8N_WEBHOOK_URL" ]; then
     N8N_HOST=$(grep "^N8N_HOST" .env | cut -d= -f2-)
     N8N_WEBHOOK_URL="${N8N_HOST}/webhook/6KFFgjd26nF0kNCA"
   fi
   echo "Testing webhook: $N8N_WEBHOOK_URL"
   curl -s -X POST "$N8N_WEBHOOK_URL" \
     -H "Content-Type: application/json" \
     -d '{
       "workflowId": "6KFFgjd26nF0kNCA",
       "roomId": "test-room-01",
       "jobId": "test-job-01",
       "deviceId": "test-device",
       "userText": "Merhaba, bugün ne var?",
       "messages": [],
       "systemPrompt": "Sen Coziyoo AI asistanısın.",
       "locale": "tr"
     }' | python3 -m json.tool
   ```

5. Confirm the response body contains a `replyText` key with a non-empty string.

**Verify:** The curl command returns a JSON object with `"replyText": "..."` in under 60 seconds. If it times out or returns `{ "executionId": ... }` instead, the `respondToWebhook` node is not firing — re-check the workflow in n8n UI.

### Wave 2 — Live end-to-end session test

> Prerequisite: Wave 1 pre-flight checks all passed.

#### Task 04-04-03: Single-turn live session test

**What:** Start a real voice session from the mobile app, speak one sentence, and confirm the agent speaks a response.

**Steps:**

1. Ensure all services are running:
   - API: `npm run dev:api` (or production systemd service)
   - Voice agent worker: `python -m voice_agent.entrypoint`
   - Voice agent join API: `uvicorn voice_agent.join_api:app --port 9000`
   - n8n: running and workflow active (confirmed in Wave 1)

2. Open the mobile app on a physical device (iOS or Android — not simulator).

3. Navigate to the voice session screen and tap **Start**.

4. Wait for "Connected" status to appear.

5. Speak clearly: "Bugün menüde ne var?" (or equivalent in the configured locale).

6. Wait for the agent to respond with synthesised speech (TTS audio in the LiveKit room).

**Expected outcomes:**
- Mobile shows "thinking" or "speaking" state after you stop talking
- Agent speaks an audible reply within 15 seconds of your utterance ending
- Voice agent log shows: `N8N response path=webhook status=200 answer=...`
- Voice agent log does NOT show: `falling back to execution API`

**Verify:**
```bash
# Check voice agent logs for the turn:
tail -50 /workspace/.runtime/voice-agent-requests.log 2>/dev/null | grep "N8N response\|path=webhook"
```
Expected: line with `path=webhook status=200 answer=<non-empty>`

#### Task 04-04-04: Multi-turn session test (conversation history)

**What:** Continue the session from Task 04-04-03 for at least 3 turns and confirm the agent maintains conversation context.

**Steps:**

1. Turn 1 (already done in Task 04-04-03): "Bugün menüde ne var?"
2. Turn 2: Ask a follow-up that requires context from Turn 1. Example: "Bundan bir tane istiyorum." (I want one of those.) The agent should reference what was discussed in Turn 1 — not give a generic unrelated response.
3. Turn 3: Ask for confirmation or a different item.

**Expected outcomes:**
- Each turn produces a spoken reply within the timeout window
- Turn 2 and Turn 3 replies demonstrate awareness of prior turns (the `messages` history is being sent and used by n8n)
- No "falling back to execution API" in voice agent logs across all 3 turns

**Verify:**
```bash
# Check voice agent logs for 3 successful webhook turns:
tail -100 /workspace/.runtime/voice-agent-requests.log 2>/dev/null | grep "path=webhook" | wc -l
# Expected: >= 3
```

Manual: Listen to the audio — Turn 2 response should reference Turn 1 content.

#### Task 04-04-05: Timing verification

**What:** Confirm n8n processes each turn and responds within the 60-second n8n webhook timeout, and within a reasonable latency for voice UX (target: under 10 seconds end-to-end for a typical short utterance).

**Steps:**

1. During the multi-turn test, note the time between when you stop speaking and when the agent starts responding (wall clock).

2. Check the voice agent request log for the `n8n request` and `N8N response` timestamps to compute n8n round-trip time.

3. If any turn exceeds 10 seconds: check the n8n workflow execution time in the n8n UI (Executions panel → select the execution → check duration per node). The `Reply LLM (Ollama)` node is the bottleneck if it is slow.

**Timing targets:**
- n8n webhook response: under 10 seconds (ideally under 5 seconds for a short LLM call)
- n8n absolute limit: 60 seconds (hard n8n timeout)

**Verify:** All turns in the multi-turn test complete within 10 seconds of utterance end. If any turn exceeds 30 seconds, flag as a performance issue (not a Phase 4 blocker — log and move on).

## Risks

- **n8n 60-second webhook timeout (Pitfall 4):** If the Ollama model inside n8n is slow or the host is under load, individual turns may time out. This is not a code bug — it is an operational issue. Mitigation: use a fast/small model for the intent step and a larger model only for the reply step in the n8n workflow, or reduce the `messages` history length sent per turn.
- **`roomId` as session key:** n8n receives `roomId` on every turn. If n8n needs to persist session state between turns (e.g., order draft), it must key on `roomId`. The voice agent's `messages` array carries the conversation history — n8n does not need separate session storage for conversation context, only for out-of-band state like a pending order draft.
- **Physical device required:** The live test must run on a real iOS or Android device; Expo simulator does not support real microphone capture in the same way. Ensure a physical device is available before scheduling this plan.
- **Workflow `UNKNOWN` confidence (now resolved):** The research noted MEDIUM confidence on whether the workflow uses `respondToWebhook`. Task 04-04-01 confirms it does. If for any reason the deployed n8n instance has an older version of the workflow without this node, activate the latest exported JSON from `apps/voice-agent/workflows/brain_6KFFgjd26nF0kNCA.json` in the n8n UI.
