# Plan 04-03: Wire Voice Agent to N8N Webhook (Respond to Webhook)

## Goal

The voice agent calls the n8n webhook as the sole path for per-turn LLM responses; the execution API fallback is demoted to a clearly-logged last resort that surfaces failures instead of hiding them; and the n8n workflow is confirmed to use the `respondToWebhook` node so `replyText` is returned inline.

## Context

**Good news from workflow inspection:** `brain_6KFFgjd26nF0kNCA.json` already has a `respondToWebhook` node as its last node (`n8n-nodes-base.respondToWebhook | Respond`). The n8n workflow side is already correctly configured.

**The remaining problem:** The voice agent's `_run()` method (entrypoint.py lines 588–624) silently falls back to `_run_execution_api` whenever the webhook call raises any exception. The execution API fallback (`_run_execution_api`, lines 655–720) polls exactly once after a 600ms sleep — unreliable for per-turn latency, and it masks webhook misconfigurations with a confusing "execution result missing answer text" error.

The fix: when the webhook path raises `APIStatusError` (an HTTP error from n8n, meaning n8n was reachable but rejected the request), do not fall back — re-raise immediately. When the webhook raises `APIConnectionError` (network-level failure, meaning n8n was not reached), log a clear warning before falling back, so operators see in logs which path is being used.

Additionally, add explicit error logging when `_extract_n8n_answer` finds no answer text in an n8n response, to help diagnose the "n8n returned 200 but no answer" scenario caused by workflows that do not have a `respondToWebhook` node.

## Requirements Addressed

| Req ID | Description |
|--------|-------------|
| TURN-01 | Voice agent sends each user turn to n8n with session ID; n8n maintains session state across turns |
| TURN-03 | N8N workflow uses "Respond to Webhook" node and returns a reply before the 60-second timeout |
| TURN-04 | Voice agent correctly parses `{ "replyText": "..." }` from n8n response and speaks it back |

#### Task 04-03-02: Demote execution API fallback and add explicit failure logging

**File:** `apps/voice-agent/src/voice_agent/entrypoint.py`

**What:** In the `_run()` method (lines 588–624), change the fallback logic so that:

1. If the webhook raises `APIStatusError` (n8n returned an HTTP error status), re-raise immediately without falling back. An HTTP error from n8n means the webhook was reached but rejected — falling back to the execution API won't help and will produce a confusing error.
2. If the webhook raises `APIConnectionError` (connection failed, timeout, or empty answer), log a `logger.warning` at `ERROR` level that names the fallback path before proceeding to it.
3. Prepend a comment block to `_run_execution_api` documenting it as a last-resort fallback that should not be relied on for production per-turn flow.

**Why:** Currently any exception from `_run_webhook` silently triggers the fallback. When n8n is misconfigured (wrong URL, wrong auth), this produces a secondary and even less informative error from the execution API, making the real cause hard to find in logs.

**How:**

Replace the fallback block in `_run()` (lines 599–624) with:

```python
        except APIStatusError as exc:
            # n8n was reachable but returned an HTTP error — do not fall back,
            # re-raise so the caller sees the real HTTP status from n8n.
            n8n_request_logger.error(
                "N8N response path=webhook http_error=%s — not falling back to execution API",
                exc,
            )
            raise
        except Exception as exc:
            webhook_error = exc
            n8n_request_logger.error(
                "N8N response path=webhook error=%s — falling back to execution API (last resort, unreliable for per-turn flow)",
                exc,
            )

        # Execution API fallback: single-poll, 600ms delay, requires N8N_API_KEY.
        # This path is unreliable for per-turn latency. It exists only to provide
        # a secondary attempt when the webhook path fails at the network level.
        # Do NOT rely on this path for production. Fix the webhook configuration instead.
        try:
            answer = await self._run_execution_api(session=session, payload=payload)
            n8n_request_logger.info("N8N response path=execution_api status=200 answer=%s", answer)
            ...
```

Keep the rest of the execution API handling (the `_event_ch.send_nowait` call and error re-raises) unchanged.

Also add this header comment above the `_run_execution_api` method definition (line 655):
```python
    # ── Execution API fallback (last resort) ─────────────────────────────────────
    # This path is single-poll with a fixed 600ms sleep. It requires N8N_API_KEY
    # and is not reliable for per-turn latency. Use only when webhook path is
    # unavailable at the network level.
    async def _run_execution_api(self, *, session: aiohttp.ClientSession, payload: dict[str, Any]) -> str:
```

#### Task 04-03-03: Add explicit logging when n8n returns empty answer shape

**File:** `apps/voice-agent/src/voice_agent/entrypoint.py`

**What:** In `_run_webhook` (line 649–652), before raising `APIConnectionError("n8n webhook returned empty answer")`, add a `logger.error` that includes the raw response body (truncated to 500 chars). This helps distinguish "n8n returned 200 with an unexpected JSON shape" from "n8n returned 200 with a valid shape that just had no text".

**Why:** The most common misconfiguration for operators is running a workflow without a `respondToWebhook` node — n8n responds with `{ "executionId": "..." }` or an empty `{}`. Currently the error message "n8n webhook returned empty answer" gives no hint about what was actually returned. The research confirms `_extract_n8n_answer` and `_deep_find_answer` are robust — if both return empty, the body itself is the diagnostic signal.

**How:**

Current lines 648–652:
```python
            answer = _extract_n8n_answer(parsed)
            if not answer:
                answer = _deep_find_answer(parsed)
            if not answer:
                raise APIConnectionError("n8n webhook returned empty answer", retryable=False)
```

Replace with:
```python
            answer = _extract_n8n_answer(parsed)
            if not answer:
                answer = _deep_find_answer(parsed)
            if not answer:
                n8n_request_logger.error(
                    "N8N webhook returned 200 but no answer text found. "
                    "Check that the n8n workflow has a 'Respond to Webhook' node that returns {replyText}. "
                    "Raw response (first 500 chars): %s",
                    str(parsed)[:500],
                )
                raise APIConnectionError("n8n webhook returned empty answer", retryable=False)
```

## Risks

- **Removing silent fallback may expose previously hidden webhook failures in production.** This is intentional — silent failures mean the agent appears to work while actually using the unreliable execution API path. The new error logs are the correct signal.
- **The 60-second n8n webhook timeout (Pitfall 4 from research) is not addressed here.** The workflow already has a `respondToWebhook` node; if the Ollama LLM call inside n8n is slow, the timeout will surface as an `APIStatusError` (HTTP 408/504) or a connection timeout. Keep the LLM model fast. This is an operational concern, not a code bug.
- **`APIStatusError` import:** Confirm this class is imported in `entrypoint.py`. Per the research it is used on line 641 — it is already imported. The new `except APIStatusError` block will work without additional imports.
