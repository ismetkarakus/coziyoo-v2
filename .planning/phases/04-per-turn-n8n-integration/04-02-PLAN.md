# Plan 04-02: Consolidate N8N Webhook URL Resolution

## Goal

The n8n LLM webhook URL is resolved through exactly one code path — the API's `resolveProviders` — and the voice agent uses only what the API sends in job metadata, without independently reading competing env vars that can produce a different URL.

## Context

There are currently two independent URL resolution paths that can produce different results:

**Path A (API):** `apps/api/src/services/resolve-providers.ts` lines 141–161 builds an `n8n` object with `baseUrl`, `workflowId`, `webhookUrl`, `webhookPath`, etc. This object is serialised into `agentMetadata` JSON and dispatched to the voice agent via the LiveKit job metadata field. The API does not read `N8N_LLM_WEBHOOK_URL` at all.

**Path B (voice agent):** `apps/voice-agent/src/voice_agent/entrypoint.py` lines 441–477, function `_resolve_n8n_webhook`, takes the values from Path A as inputs but also reads two env vars independently:
- `os.getenv("N8N_LLM_WEBHOOK_URL", "")` at line 447 — if set in the voice agent's environment, this overrides everything the API sent
- `os.getenv("N8N_LLM_WEBHOOK_PATH", "")` at line 451 — used as the webhook path if the API sent no path

If `N8N_LLM_WEBHOOK_URL` is set in the voice agent's env but not the API's env, the two sides use different URLs. There is no log message at worker startup showing what URL will be used — it only logs at job start.

The fix has two parts: (1) add `N8N_LLM_WEBHOOK_URL` to the API's env schema so the API can pass an explicit absolute URL through job metadata, and (2) remove the voice agent's independent `os.getenv` reads of these two vars from `_resolve_n8n_webhook`, making the function use only the values received from the API. Add startup diagnostic logging on both sides.

| Req ID | Description |
|--------|-------------|

## Tasks

### Wave 1 — Add N8N_LLM_WEBHOOK_URL to API env schema and pass it through resolveProviders

> Prerequisite: None.

#### Task 04-02-01: Add N8N_LLM_WEBHOOK_URL to API env schema and wire through resolveProviders

**Files:**
- `apps/api/src/config/env.ts`
- `apps/api/src/services/resolve-providers.ts`

**What:** Add `N8N_LLM_WEBHOOK_URL` as an optional URL field in the API's Zod env schema. Then, in `resolveProviders`, use it as the `webhookUrl` value (highest priority) in the `n8n` object returned.

**Why:** The voice agent's `_resolve_n8n_webhook` already gives `webhook_url` parameter (from `providers.n8n.webhookUrl`) first priority at line 447. Once the API passes the value through, setting a single `N8N_LLM_WEBHOOK_URL` in the shared root `.env` will produce the same resolved URL on both sides.

**How:**

In `apps/api/src/config/env.ts`, add after line 83 (`N8N_LLM_WEBHOOK_PATH`):
```typescript
N8N_LLM_WEBHOOK_URL: z.string().url().optional(),
```

In `apps/api/src/services/resolve-providers.ts`, update the `webhookUrl` field in both branches of the n8n resolution (the `defaultN8nServer` branch, line 146, and the legacy branch, line 156):

```typescript
// defaultN8nServer branch — replace lines 146-148:
webhookUrl:
  strOrNull(env.N8N_LLM_WEBHOOK_URL) ??
  strOrNull(defaultN8nServer.webhookUrl ?? defaultN8nServer.endpoint ?? defaultN8nServer.url) ??
  null,

// legacy branch — replace lines 156-158:
webhookUrl:
  strOrNull(env.N8N_LLM_WEBHOOK_URL) ??
  strOrNull(legacyN8n.webhookUrl ?? legacyN8n.endpoint ?? legacyN8n.url) ??
  null,
```

`env.N8N_LLM_WEBHOOK_URL` takes absolute priority because an explicit absolute URL in the environment is the clearest possible configuration signal.

**Verify:**
```bash
npm run build:api 2>&1 | grep -c "error" && echo "no TS errors" || echo "TS errors found"
# Spot-check: confirm N8N_LLM_WEBHOOK_URL appears in the env schema
grep "N8N_LLM_WEBHOOK_URL" apps/api/src/config/env.ts
grep "N8N_LLM_WEBHOOK_URL" apps/api/src/services/resolve-providers.ts
```

### Wave 2 — Remove competing env var reads from voice agent and add diagnostic logging

> Prerequisite: Wave 1 must be complete. The voice agent's `_resolve_n8n_webhook` changes depend on knowing that the API now passes `webhookUrl` explicitly.

#### Task 04-02-02: Remove independent env var reads from _resolve_n8n_webhook

**File:** `apps/voice-agent/src/voice_agent/entrypoint.py`

**What:** In `_resolve_n8n_webhook` (lines 441–477), remove the two independent `os.getenv` reads. The function's `webhook_url` and `webhook_path` parameters already receive the API-resolved values — they should be the only inputs.

**Why:** If `os.getenv("N8N_LLM_WEBHOOK_URL", "")` is left in the function, setting this env var on the voice agent host still overrides the API's choice — exactly the split-brain scenario we are eliminating.

**How:**

Current line 447:
```python
explicit = _normalize_base_url(webhook_url or os.getenv("N8N_LLM_WEBHOOK_URL", ""))
```
Replace with:
```python
explicit = _normalize_base_url(webhook_url)
```

Current line 451:
```python
raw_path = (webhook_path or "").strip() or (os.getenv("N8N_LLM_WEBHOOK_PATH", "") or "").strip()
```
Replace with:
```python
raw_path = (webhook_path or "").strip()
```

No other lines in `_resolve_n8n_webhook` read env vars. The `os.getenv("N8N_HOST")` used in `_build_llm` (called before `_resolve_n8n_webhook`) is a fallback for `n8n_base_url` when `providers.n8n.baseUrl` is null — this is acceptable and should remain since the API also falls back to `env.N8N_HOST` in `resolveProviders`.

**Verify:**
```bash
grep "N8N_LLM_WEBHOOK_URL\|N8N_LLM_WEBHOOK_PATH" apps/voice-agent/src/voice_agent/entrypoint.py
# Expected: zero matches after the change
```

#### Task 04-02-03: Add startup diagnostic logging for resolved webhook URL

**Files:**
- `apps/voice-agent/src/voice_agent/entrypoint.py` (worker startup)
- `apps/api/src/server.ts` (or `apps/api/src/config/env.ts` — see how below)

**What:** Log the resolved n8n webhook URL at startup on both sides. On the API side, log the configured `N8N_HOST` and `N8N_LLM_WEBHOOK_URL` values at server start. On the voice agent side, log the resolved URL once when a job starts, immediately after `_resolve_n8n_webhook` returns, before any HTTP calls.

**Why:** Neither side currently logs the webhook URL in a way operators can find at startup. The voice agent logs it in `_build_llm` but only after a job is dispatched. An operator debugging "why is my webhook call going to the wrong URL" needs to see this in startup/boot logs.

**How:**

In `apps/api/src/server.ts`, after the API starts listening, add:
```typescript
console.log("n8n config: host=%s webhookUrl=%s webhookPath=%s workflowId=%s",
  env.N8N_HOST ?? "(not set)",
  env.N8N_LLM_WEBHOOK_URL ?? "(not set)",
  env.N8N_LLM_WEBHOOK_PATH || "(default: /webhook/{workflowId})",
  env.N8N_LLM_WORKFLOW_ID,
);
```

In `apps/voice-agent/src/voice_agent/entrypoint.py`, in the `_build_llm` function, locate the existing log call that mentions the webhook URL (research confirms it logs `"Using N8N LLM webhook: %s workflow=%s"`). Verify this log is at `logger.info` level (not `logger.debug`). If it is at `debug` level, change it to `info`. No new log line needed if the existing one is already at `info`.

**Verify:**
```bash
# API startup log should contain "n8n config:"
npm run dev:api 2>&1 | grep "n8n config" | head -3

# Voice agent: the existing "Using N8N LLM webhook" log line should appear at info level in job logs
grep -n "Using N8N LLM webhook" apps/voice-agent/src/voice_agent/entrypoint.py
```

## Acceptance Criteria

- [ ] `N8N_LLM_WEBHOOK_URL` is a recognized env var in the API's Zod schema (optional, validated as URL)
- [ ] When `N8N_LLM_WEBHOOK_URL` is set in the root `.env`, `resolveProviders` returns it as `n8n.webhookUrl`, and the voice agent's `_resolve_n8n_webhook` uses it as the final URL
- [ ] `_resolve_n8n_webhook` no longer reads `os.getenv("N8N_LLM_WEBHOOK_URL")` or `os.getenv("N8N_LLM_WEBHOOK_PATH")` independently
- [ ] API logs the n8n `N8N_HOST`, `N8N_LLM_WEBHOOK_URL`, and `N8N_LLM_WORKFLOW_ID` values at startup
- [ ] `npm run build:api` completes without TypeScript errors
- [ ] `grep "N8N_LLM_WEBHOOK_URL\|N8N_LLM_WEBHOOK_PATH" apps/voice-agent/src/voice_agent/entrypoint.py` returns zero matches

## Risks

- **Operators using `N8N_LLM_WEBHOOK_URL` in the voice agent's env today will lose that override after this change.** This is intentional — they should move the value to the root `.env` so both sides see it. Document in the PR description.
- **The `os.getenv("N8N_HOST")` fallback in `_build_llm` is not removed.** This is intentional: `N8N_HOST` is an API-side env var, but operators may also set it on the voice agent host as a convenience. Removing it is out of scope and would break configurations that rely on it as the only n8n setting.
- **`resolve-providers.ts` type for `webhookUrl` is already `string | null`** (line 26 of the file) — no type change needed; `strOrNull(env.N8N_LLM_WEBHOOK_URL)` returns `string | null` correctly.
