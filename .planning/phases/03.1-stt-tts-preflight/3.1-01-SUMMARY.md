# Phase 3.1 Plan 01: STT/TTS Preflight Health Check Summary

**One-liner:** Added server-side STT and TTS reachability checks to the session dispatch preflight, surfaced as typed error codes that mobile renders as specific user-facing warnings.

## What Was Done

**API (`apps/api/src/routes/livekit.ts`):**
1. Added `probeServiceHealth(baseUrl)` function — does a GET with 3s timeout; any HTTP response = reachable, connection error/timeout = unreachable.
2. In `/starter/session/start`, after `resolveProviders(settings)` and before n8n preflight:
   - If `sttEnabled !== false` and `resolved.stt.baseUrl` is set → probe STT; return `STT_UNAVAILABLE` (503) if down.
   - If `ttsEnabled !== false` and `resolved.tts.baseUrl` is set → probe TTS; return `TTS_UNAVAILABLE` (503) if down.

**Mobile (`apps/mobile/src/screens/HomeScreen.tsx`):**
1. In `resolveStartSessionError`, added handling for `STT_UNAVAILABLE` → "Speech recognition unavailable. Please check the STT server and try again."
2. Added handling for `TTS_UNAVAILABLE` → "Voice synthesis unavailable. Please check the TTS server and try again."
